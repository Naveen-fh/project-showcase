1 Big Picture — What Is This System?

This chatbot is a AI-primary customer-support system for a food delivery platform (Foodhub / CustomerApp). It is NOT a simple ChatGPT wrapper — but it is also not a simple rule engine. The AI runs first on every message to classify intent and route the conversation. Guardrail rules (keyword/regex) exist as a fallback only — used when the AI is unavailable, times out, or returns low confidence.

Think of it this way: the AI is the brain that decides what to do. Guardrails are the safety net underneath it. RAG is a fully-built search engine that is ready to be wired in but is not yet active in the live message pipeline.

3 Full Message Flow — User Types to Bot Replies

Walk-through of exactly what happens when a customer sends "I didn't receive my pizza", from the moment they hit send to the final bot reply or live agent handover.

  • 1
    Customer sends message
    App sends POST /chat/sessions/:chatId/messages with { text: "I didn't receive my pizza" }
  • 2
    Security & early gate checks (before AI)
    scanText() in chatSecurity.js scans for jailbreak attempts, abuse patterns, and out-of-scope requests. The middleware also applies rate limiting (rateLimiter) and input validation. CXO flow and guest-order gates are also checked here. This is pure logic — no AI, no DB — so it's instant.
  • 3
    AI multi-intent detection
    aiDetectMultipleIntents(text) — the AI checks whether the customer mentioned more than one issue in one message (e.g. "my order is late AND an item is missing"). If so, each issue is queued separately. Falls back to a regex version if AI is unavailable.
  • 4
    AI main router — PRIMARY classification
    aiRouteMessage(text, apiKey) is called. This sends the message to the LLM with a system prompt that includes all known flow types, the customer snapshot, order history, and RAG context (vector search results). The AI returns: { flowType: "complaint", issueKind: "order_not_arrived", confidence: "high" }.
  • 5a
    AI confident → AI-handled route
    If confidence !== 'low', the AI result is used directly. The backend routes to the correct handler — e.g. flowType = 'complaint' triggers the triage system. No regex matching needed. _routerReturnedEarly = true and the fallback chain is skipped entirely.
  • 5b
    AI unavailable or low confidence → Regex fallback
    Only if aiRouteMessage() fails (API timeout, network error) or returns confidence: 'low' do the guardrail matchers run: matchOrderComplaintIntent(), matchDeliveryIntent(), etc. These are the keyword/regex rules stored in Supabase.
  • 6
    Triage kicks in (complaint path)
    orderComplaintTriage.js shows an order picker (only delivered/cancelled orders as chips). Customer taps: "#1234 Franco's — £12.50 — Delivered"
  • 7
    Diagnostic questions
    Bot asks: "Was the order handed to you or left outside?"
  • 8
    Resolution chips
    [ Request Refund ]   [ Request Replacement ]   [ Speak to Agent ]   [ I'm sorted ]
  • 9a
    Self-serve resolution
    buildRefundSelfServeMessage() generates advice. May connect to Zoho if severity is high.
  • 9b
    Escalation to live agent
    POST /zoho/handover → creates Zoho conversation → sends full triage context note → agent takes over.

4 Guardrails — The Fallback Rule Engine

Guardrails are not the first thing that runs — the AI is. Guardrails are the fallback safety net that activates only when the AI is unavailable (timeout, API down) or returns low-confidence results. They are stored in Supabase per flow and edited via the admin UI at localhost:4010.

Despite being a fallback, guardrails are critically important. If the AI API goes down, the chatbot continues to function — it just uses fast keyword/regex matching instead. The rules in Supabase define the bot's deterministic behaviour when the AI can't be trusted or reached. They are also used to inform the AI's system prompt via system_prompt_summary fields, so even when the AI is running, the guardrail data shapes how it responds.

What a Guardrail Looks Like

5 RAG — Built, Then Removed (Historical)

Status: REMOVED from the codebase Everything below describes infrastructure that used to exist. chatbot/helpers/ragHelper.js has been deleted, along with every call site that referenced it. A repo-wide grep for ragHelper, buildRagContext, findEscalationIntent, vectorSearchService, embeddingService, and faq_documents returns zero hits. This section is kept for historical context — do not build against any of it.

RAG stood for Retrieval Augmented Generation. The idea was to give the AI relevant facts from a knowledge base before it answered, so it wouldn't hallucinate or make up platform-specific policies. The infrastructure below was fully built but never called from the live message pipeline — and has since been deleted rather than activated.

What Replaced It

The flexibility RAG was meant to provide (letting message copy vary by context without hardcoding every variant) is now handled by two simpler, already-shipped mechanisms instead:

  • Per-flow guardrail JSON — /gaurdrules/:flow still exists and is fetched by the app for every flow (account, delivery, genralenquiry, menu, order_complaint, order_modification, order_status, promotions, refunds, payments, edgecases, voucher, delivery_address, static_info) — but it now returns plain rule JSON only, no vector search results merged in.
  • Backend-configurable message templates — see §22. Instead of semantic search picking the "right" phrasing, the backend guardrail JSON's meta.refundTemplates can directly override message copy per region/flag, resolved client-side by resolveRefundTemplates() in refundsFlow.js.

What RAG Would Solve

If you ask an AI "Can I modify my order?", it has no idea about this platform's specific rules. RAG would fix this by running a semantic search on the knowledge base first and injecting the matching policy into the AI's system prompt. The AI then answers based on actual platform rules rather than general training data.

Without RAG, the AI's accuracy depends entirely on how well the system prompt and guardrail summaries are written by a human. With RAG, the AI automatically pulls the most relevant context from a database of thousands of intents and FAQs — it scales without manual prompt engineering.

8 Zoho SalesIQ — Why We Use It & How

What Problem Zoho API Solves

The chatbot is designed to handle the majority of customer queries automatically — order status checks, delivery tracking, menu questions, account changes. But there is a class of situations where a human agent must be involved: food poisoning and allergy complaints (legal/safety risk), refund disputes over a certain value, customers who are clearly distressed or angry, and any situation where the AI reaches its confidence limit and marks the conversation for handover.

The question is: where does that human conversation happen? The support team at Foodhub uses Zoho SalesIQ as their day-to-day live chat platform. Agents are already logged in there, queues are managed there, supervisors monitor it there. Building a separate live chat system inside our own app would duplicate all of that — agent dashboards, queue management, availability tracking, SLA reporting, read receipts, typing indicators — a months-long build for something that already exists.

Instead, the chatbot's backend uses the Zoho SalesIQ API to programmatically create a conversation inside Zoho on behalf of the customer, transfer all the triage context the bot has collected, and then step back. The agent sees the customer arrive in their normal Zoho queue with a detailed briefing note — they don't need to ask the customer to explain everything again.

9 Zoho OAuth & Refresh Token — Why and How

Zoho API requires OAuth2 — you cannot use a username/password. Instead you get two tokens:

Access Token
Short-lived (1 hour). Used in every API call header: Authorization: Zoho-oauthtoken ...
Refresh Token
Long-lived (doesn't expire unless revoked). Used to get new access tokens. Stored in .env. One-time setup.

Supabase is a managed PostgreSQL database. It adds auth, storage, and edge functions. It also supports pgvector — the PostgreSQL extension for semantic / vector search — but see the note below before assuming it's in use.

pgvector is no longer used The section below describes how vector search worked when RAG existed (see §5). RAG has since been removed — the app no longer generates embeddings, queries knowledge_embeddings/faq_documents, or calls pgvector at all. Kept here for historical reference only.

How pgvector Worked (Historical — RAG Removed)

  • 1
    A new FAQ or intent is added to the knowledge base.
  • 2
    Embedding service sends the text to OpenAI → gets back ~1536 numbers (a vector representing meaning).
  • 3
    Vector stored in knowledge_embeddings table alongside the original text and metadata.
  • 4
    User asks a question → question also converted to a vector.
  • 5
    Supabase runs a SQL query finding stored vectors with smallest "distance" (most similar meaning).
  • 6
    Top matches returned → become your RAG context.
Service Key vs Anon Key The backend uses the service key (full DB access, bypasses RLS). Never expose this client-side. The anon key is for safe public access only.

18 Redis / ElastiCache — Session Persistence Layer

Two classes of data need to survive app refreshes and device switches: triage context (which order was selected, packaging answers, resolution choices) and live-agent session IDs (Zoho conversationId + visitorId). Both are stored in Redis / AWS ElastiCache keyed by convId with an 8-hour TTL.

Why not Supabase for this? Supabase (PostgreSQL) is used for durable records — chat sessions, messages, guardrail rules. Redis is used for short-lived, high-frequency read/write state that needs to survive server restarts but not forever. Session context and live-agent IDs are only relevant for the duration of an active conversation (~8 hours max).

19 AI Rate Limiting & Session ID Threading

All AI calls in chatHelpers.js route through a backend cloud function (CHAT_CLOUD_FUNCTION_URL). The backend applies per-session rate limiting using the sessionId sent with every request. When the AI service returns HTTP 429 (too many requests), the app distinguishes this from a regular failure and handles it gracefully instead of falling through to the guardrail fallback.

The AI_RATE_LIMITED Sentinel

/** Returned when the AI service is rate-limited (429). Not a fatal failure — show retry message. */
export const AI_RATE_LIMITED = Object.freeze({ __aiRateLimited: true });

// All AI call sites:
if (res.status === 429) {
    const e = new Error('Rate limited');
    e.isRateLimit = true;
    throw e;
}
// In catch block:
return e?.isRateLimit ? AI_RATE_LIMITED : AI_FAILED;
Return ValueMeaningWhat the App Does
AI_RATE_LIMITEDBackend returned 429 — AI is overloaded, not brokenShows user-facing "AI is busy" message. Does NOT fall back to guardrail regex — the AI is available, just rate limited.
AI_FAILEDNetwork error, timeout, or non-429 server errorFalls back to guardrail regex matching as normal.

Session ID Threading

Every AI request includes the active chat session ID. The backend uses this for per-session request tracking in Redis/ElastiCache — ensuring one user can't accidentally exhaust the AI quota for all users.

// chatHelpers.js
let _activeChatSessionId = '';
export function setActiveChatSessionId(id) { _activeChatSessionId = id || ''; }

// Every AI call body:
body: JSON.stringify({
    messages: msgs,
    sessionId: _activeChatSessionId || undefined,
    max_tokens: 40
})

Why This Architecture

No API key in app bundle
CHAT_OPENAI_API_KEY removed from CustomerWeb and FranchiseWeb env files. Auth lives in the backend cloud function only.
Per-session fairness
sessionId lets the backend cap requests per active conversation, preventing one session from monopolising the AI quota.
Graceful 429 handling
Rate limiting is treated as a temporary delay, not a failure. The user sees a message and can retry — the guardrail fallback is not triggered.

AI Topic-Shift Confirmation — aiConfirmTopicShift()

When the bot detects the customer is switching topic mid-flow, it asks a clarifying question: "Do you want to switch to X, or carry on with Y?". The reply is then classified by a dedicated AI helper rather than by regex.

aiConfirmTopicShift(userText, { newTopicLabel, currentFlowHint }, apiKey, signal)
// Returns one of:
//   'switch'      — customer confirmed they want the new topic
//   'continue'    — customer wants to stay with the current flow
//   'other_flow'  — customer asked about something completely unrelated (high confidence)
//   'unclear'     — reply too ambiguous — bot re-asks the question

// Fallback on any error: 'unclear'
Edge case: ambiguous "ok" or "sure" The AI prompt is biased toward 'switch' over 'other_flow' when uncertain. Only returns 'other_flow' when the customer's message is clearly off-topic for both options (e.g. asking about a voucher when the choice was delivery vs. missing item).

21 Global Message Middleware — Question De-Duplication

New file: chatbot/helpers/globalMessageMiddleware.js. Despite the name, this is not a request router — it doesn't decide which flow handles a message. It's a session-memory / anti-repetition layer that stops the bot from asking the same triage question twice and remembers the answer once given.

Why it exists Before this, a customer could get asked "was the order handed to you or left outside?" more than once in the same conversation — after a reload, a topic detour, or an AI misclassification. This middleware makes that impossible past 2 asks, and skips the question entirely once it's already been answered.
