🧠 How the Chatbot Works
Before diving into each flow, here is how the chatbot decides what to do with every customer message.
1
Customer sends a message
The customer types anything — "where is my order", "I want a refund", "hi", etc.
2
Security & token budget check
Before the AI is called, the message is scanned for attack patterns (prompt injection, XSS, SQL injection, etc.) and the session's token usage is checked against the 30,000-token budget. If either check fails, the message is blocked and an amber "AI agents busy" banner appears — with a direct link to contact live agents. Live agent requests always bypass this check.
3
AI reads the message and classifies the intent
The AI figures out what the customer wants and assigns a confidence score. It also checks which order the customer is talking about.
4
Chatbot picks the right flow
Based on the intent, the chatbot routes the customer into the correct flow (e.g. complaint flow, refund flow, order status flow).
5
Chatbot asks questions and gathers information
The flow asks the customer relevant questions (e.g. which order, which items, how delivered) to fully understand the issue.
6
Resolve or escalate
The chatbot either resolves the issue automatically (e.g. issue a refund, provide info) or connects the customer to a live agent.
ℹ️
Topic switching: If a customer is mid-flow and suddenly asks about something different, the chatbot detects this and asks "You seem to be asking about something new — would you like to switch topics?" before changing flow. This prevents accidental flow resets.

🛡️ Guardrail Flows (Backend Rules)
These are structured rule sets stored in the backend database. Each flow has its own set of intents and guardrails — the chatbot matches the customer's message against these rules to stay on-topic and give consistent answers within each flow.
💡
What are guardrail flows?
Think of guardrails as a rulebook for each topic. When a customer is in the "Refunds" flow, the guardrails make sure the chatbot only discusses refund-related things — it won't drift into unrelated topics. Each flow has its own table in the backend that the ops/admin team can update.

General Enquiry genralenquiry General Q&A about the platform

ℹ️
How it connects to the flows above: Every detailed flow in this document (order complaint, refunds, etc.) is backed by one of these guardrail tables. The table controls what questions the chatbot is allowed to answer within that topic. If a question falls outside the guardrail, the chatbot either redirects or escalates to a live agent.
🎭 AI Personality & Tone Rules
The chatbot is governed by a 10-rule personality block embedded in every system prompt. These rules make it feel like a real support person — not a scripted bot.
🎭

Core Personality Rules

Applied to every message the AI generates — cannot be overridden by customer input

1
Empathy first, action second
Acknowledge the situation before jumping to solutions. A person feels heard before they feel helped.
2
No hollow affirmations
Words like "Certainly!", "Absolutely!", "Of course!", "Great question!", "Happy to help!" are banned. They signal a bot, not a person.
3
Short, direct replies
Responses are kept concise. No multi-paragraph essays. One clear point per message whenever possible.
4
Varied openings — no "Thanks —" starters
"Thanks —" openers are removed throughout the codebase. Replies don't start with the same phrase twice in a row. Dynamic AI generation is used to vary message openers.
5
Human closers
Messages end naturally — not always with "Let me know if there's anything else I can help with!" (which is banned as a filler closer). The AI uses context-specific endings.
6
Use the customer's name naturally
When a name is available, it's used occasionally — not in every message, which would feel unnatural.
7
"Sorry" is used with intent — not as filler
"I'm sorry" is reserved for situations where something genuinely went wrong (safety, serious complaints). Routine messages don't open with apologies. "Frustrating" is a banned word.
8
Context-aware live agent & re-entry messages
When connecting to a live agent, the wait message is generated with full context — the order number, restaurant name, and the type of issue. When a customer returns from a live agent session, the welcome-back message references what they were just dealing with, not a generic opener.
💡
Quick-reply chips also follow these rules — they are written as natural phrases a real person would type, not button labels like "Yes / No / Cancel".
🚫

Banned Phrases & Words

Hard-coded exclusions across all AI-generated text

Banned phrase / word Why What to use instead
"Certainly!", "Absolutely!", "Of course!" Hollow affirmations — signal bot, not person Jump straight to the reply
"Thanks —" (as opener) Repetitive and filler Varied context-first openings
"I'm really sorry" (as generic opener) Dilutes genuine apology Reserve for safety/serious complaints only
"frustrating" Feels patronising Describe the specific issue directly
"Let me know if there's anything else I can help with!" Robotic filler closer Context-specific endings or nothing
"Happy to help!" Hollow affirmation Jump straight to the reply
🔒 Security & Rate Limits
Two layers of protection run on every message — a client-side scan and a backend middleware check. When limits are hit, customers see an amber banner rather than a broken experience.
⚠️
What the customer sees when limits are hit
An amber "AI agents busy" banner appears at the top of the chat. It includes a "Contact live agents directly" link so the customer is never left stuck. The text input is disabled until the session resets.
🛡️

Two-Layer Security Model

1
Client-side attack scan (before API call)
35 attack patterns checked on the device before anything is sent to the server. Covers: prompt injection, jailbreak attempts, XSS, SQL injection, SSRF, data exfiltration commands.
Clean → proceed to AI Attack detected → block + amber banner
2
Token budget check (per session)
Each session has a 30,000-token budget. Token usage is estimated before each AI call and tracked after each response. Approaching or exceeding the limit triggers the amber banner.
Under budget → proceed Budget exceeded → amber banner shown
3
Backend rate limiter (server middleware)
The API server enforces 30 requests per minute per session/IP. This prevents abuse of the AI endpoint from automated tools. When the AI itself returns a rate-limited signal (AI_RATE_LIMITED), the flow falls back gracefully rather than crashing.
4
Live agent requests always bypass security checks
If a customer says "talk to a human" or clicks a live agent button — even mid-banner, even at the token limit — the live agent connection path is always available. No customer is left with zero options.
✅ Live agent path always open
🗄️

Backend Database Tables (Supabase)

What gets logged and tracked for security

Table Purpose

🚨 Order Complaint Flow
The main complaint flow. Handles food problems, missing items, wrong orders, spoiled food, illness, allergies, and payment issues. All complaint sub-types go through a structured triage to gather the right information before resolving.
⚠️

All complaints go through a step-by-step triage. The specific steps depend on the complaint type. The chatbot collects just enough information to make a fair decision — then offers a resolution.

2
Context-aware wait message shown

When a customer returns to the AI chatbot after a live agent session, the welcome-back message references what they were dealing with — it does not use a generic "Welcome back, how can I help?" opener. The AI generates a relevant, context-specific greeting based on the session issue history.

Live agent requests always bypass security checks. Even if the token budget is exceeded or the AI is rate-limited, the live agent connection path is always available to the customer.

Multiple Issues (Multi-Intent Queue)

Customer raises more than one problem at the same time

Examples of multi-intent messages…
"My order is late AND my voucher didn't work" "Missing items and I want a refund" "Item was wrong, also I have a question about my account"
1
Chatbot detects multiple issues
AI identifies all the different problems mentioned and puts them in a queue.
Issue 1: Late delivery
→
Issue 2: Voucher not working
→
Issue 3: Refund request
2
Handle Issue 1 completely
Full flow for the first issue runs to completion — resolution offered or live agent connected.
3
Ask if customer wants to address the next issue
Chatbot says: "We've sorted your delivery concern — would you also like help with your voucher?" Customer confirms, and the next flow starts.
4
Each issue handled in turn
All issues in the queue are resolved one by one. If a live agent is needed mid-queue, the handoff is either done immediately or deferred until the current flow completes.
💡
Deferred live agent handoff: If the customer asks for a live agent while mid-way through another flow, the chatbot says "I'll connect you to a live agent after we've handled your current concern" — then does the handoff once the current flow finishes.
